## EchoMind 普通聊天管道阶段总结

EchoMind 的普通聊天采用"异步消息 + 顺序管线"架构。一次用户请求从前端发出到最终回复推送，会经过四个大的层次：**网关层**（接入与治理）、**消息队列层**（异步削峰）、**执行管线层**（核心 Agent 推理）、**异步记忆层**（后置写入）。下面按请求的时间顺序逐层说明。

---

### 一、网关层：接入、治理与入队

请求首先到达 `ChatController`（REST 端点 `POST /api/chat`），经过 JWT 认证后进入 `ChatApplicationService.submitAsync()`。这一步做四件事情：

**1. 请求标准化**：`ChatRequestNormalizer` 统一处理请求格式、填充默认值（agentId、sessionId、modelId 等），产出 `NormalizedChatRequest`。

**2. 治理检查（ChatGovernanceService.inspectRequest）**：这是请求进入管线前的"守门人"，执行两项关键检查。首先是用户 Token 配额校验——如果用户剩余配额不足，直接抛出 `TokenQuotaExceededException` 阻断请求。其次是敏感数据检测——对用户消息进行合规扫描，命中 BLOCK 规则时直接短路返回替代回复（不进入 Agent 管线），命中脱敏规则时返回脱敏后的消息继续流转。

**3. Token 配额预留**：治理通过后，按预估 Token 消耗量（用户消息 + 预估输出 Token）分别预留用户配额和 Provider 级预算，防止并发场景下超额使用。如果后续入队失败，这些预留会被立即释放。

**4. 发布到 RabbitMQ**：将封装好的 `ChatRequest`（含 requestId、userId、agentId、sessionId、message、modelId、traceId、附件、预留 ID 等）发布到 `echomind.chat.requests` 队列。同时注册 SSE 推送请求，客户端通过 `GET /api/chat/stream/{requestId}` 订阅后续结果。

治理通过但短路命中时，结果直接通过 `SsePushService` 推给客户端，不经过 Agent 管线。

---

### 二、消息队列层：异步消费

`ChatRabbitConsumer` 监听 `chat.requests` 队列，收到消息后调用 `ChatApplicationService.executeQueuedStream()`，最终委托给 `AgentOrchestrator` 进入执行管线。队列消费提供了削峰能力，保证高并发下 Agent 推理不会被压垮。消费失败并耗尽重试的消息进入 DLQ（死信队列），由管理端支持受控重放。

---

### 三、执行管线层：Agent 推理核心

`AgentOrchestrator` 是编排入口，它先通过 `AgentFactory` 找到目标 Agent 实例（找不到时回退到第一个可用 Agent），然后构建 `PipelineContext`（包含 userId、agentId、sessionId、userMessage、systemPrompt、modelId、附件等），最后调用 `Agent.chat(ctx)` 或 `Agent.chatStream(ctx)` 进入管线。

`ExecutionPipeline` 将所有 `PipelineStage` 按 `order()` 排序后串行执行。每个阶段都会创建 OpenTelemetry Span 进行埋点，任一阶段抛出异常时管线停止并标记失败。完整管线包含 **8 个阶段**：

#### 阶段 1：ContextEnrichStage（order=10）—— 会话上下文加载

从 Redis 加载当前会话的近期对话历史（按 `userId + sessionId` 隔离），然后把本轮用户消息追加到上下文末尾。如果记忆持久化被禁用（例如 Team 内部控制面调用），则跳过历史加载，只追加用户消息。这一步决定了模型"能看到多少轮之前的对话"。

#### 阶段 2：UserMemoryRetrievalStage（order=12）—— 用户画像与长期事实召回

分为两部分。首先从 `UserProfileSnapshotStore`（Redis）读取用户长期画像快照（压缩的用户偏好、背景、稳定约束），作为系统消息注入上下文头部。然后从 `UserMemoryStore`（Milvus 向量库）中，以用户消息为查询词做向量检索，召回 topK 条相关历史事实（按置信度排序、按最低相似度过滤），注入为"历史会话事实"系统消息。画像解决"这个用户是谁"，事实解决"这个用户以前说过什么相关的事"。查询词可由 `RetrievalQueryRewriter` 改写以提升召回精度。

#### 阶段 3：KnowledgeRetrievalStage（order=25）—— Agent 知识库检索

与会话记忆不同，知识库解决的是"这个 Agent 上传过哪些资料"。通过 `AgentKnowledgeService` 在 Milvus 中以用户消息做向量检索，召回 topK 条最相关的知识片段（带来源文件名和分块索引），以系统消息形式注入上下文。知识库按 Agent 隔离，不同 Agent 的知识不会互相污染。查询词同样支持 `RetrievalQueryRewriter` 改写。

#### 阶段 4：ModelResolutionStage（order=20）—— 模型解析

通过 `DynamicModelRouter` 将 modelId（格式为 `providerId:modelName`，如 `deepseek:deepseek-v4-flash`）解析为具体的 `ModelSpec`（包含供应商、模型名、能力标签等），写入 `PipelineContext.resolvedModel`。如果请求未指定模型，则使用 Agent 的默认模型。

> 注：虽然 order=20，但实际执行顺序在 UserMemoryRetrievalStage（order=12）和 KnowledgeRetrievalStage（order=25）之间。

#### 阶段 5：MultimodalGuardStage（order=30）—— 多模态能力校验

如果本轮消息携带图片附件，检查已解析模型是否具备 `VISION` 能力。不具备则抛出异常，防止前端绕过提示把图片发给纯文本模型。没有图片时直接放行。

#### 阶段 6：AttachmentPreparationStage（order=35）—— 附件准备

将用户上传的图片附件转换为模型可读取的格式。支持三种路径：HTTP/HTTPS URL 直接使用；OSS 存储的附件生成临时签名 URL（30 分钟有效期）；本地存储或无法生成公网 URL 的附件转为 base64 Data URL。转换结果存在 `modelAttachments` 中，不写入记忆（避免持久化大体积 URL 或 base64）。

#### 阶段 7：ResultAggregationStage（order=40）—— 结果聚合（核心）

这是整条管线的"终点"和最重要阶段，负责把前面所有准备工作汇总为一次 LLM 调用。核心流程：

1. **工具暴露**：从 `ToolRouter` 获取当前所有已启用工具（包括所有已启用 Skill 和已挂载外部 MCP 工具），作为函数列表暴露给模型。内部控制面调用可关闭工具暴露。
2. **提示词组合**：`PromptComposer` 按预算（`PromptBudget`）将系统提示、用户画像、知识库片段、对话历史、用户消息、附件等组装为完整的 `ProviderRequest`。系统提示末尾会自动追加"工具使用规则"，指导模型主动使用日期查询和联网搜索工具。
3. **治理前置检查**：`ModelInvocationPreflight` 在模型调用前做最后一次治理检查（如 Provider 级 Token 预算检查），不通过则抛出 `ModelInvocationRejectedException`。
4. **LLM 调用**：通过 `ModelProviderRegistry` 找到对应的 `ModelProvider`，支持同步调用（`chatWithUsage`）和流式调用（`streamWithUsage`）。同步模式直接返回完整响应；流式模式返回 `Flux<String>` token流，供 SSE 逐段推送。
5. **记忆决策解析**：`MemoryDecisionParser` 从模型输出中提取隐藏的记忆控制标记（决定是否异步处理用户事实和画像），并从可见文本中移除这些标记。
6. **Token 统计**：记录模型返回的 prompt/completion/total token 使用量。

#### 阶段 8：MemoryPersistStage（order=50）—— 记忆事件发布

将本轮对话内容（用户输入 → 工具调用记录 → 模型回复）按真实对话顺序封装为消息列表，通过 `ChatMemoryPersistPublisher` 发布到 RabbitMQ `chat-memory` 异步记忆写入队列。记忆决策（`MemoryDecision`）一并发布，控制后续用户事实提取行为。

> **流式模式的差异**：流式对话时，`Agent.chatStream()` 先同步执行 order≤35 的前置阶段，然后直接调用 `ResultAggregationStage.streamResponse()` 获取 token 流。`MemoryPersistStage` 不在主管线中执行，而是在 `Flux.doFinally()` 中（流成功完成后）手动调用，确保收集到完整回复后再发布记忆事件。

---

### 四、异步记忆层：后置写入

记忆写入队列 `echomind.chat-memory.persist.exchange` 采用分片消费，由 `ChatMemoryPersistConsumer` 负责：

- **MySQL**：写入完整会话历史（供前端会话列表和历史查看）。
- **Redis**：更新近期上下文窗口（供下次对话时 `ContextEnrichStage` 读取）。
- **用户事实提取**：当记忆决策要求提取时，将对话事件发送到 `echomind.user-memory.requests` 队列，由 `echomind-user-memory` 模块的轻量模型提取用户长期事实，写入 Milvus 向量库，并刷新 Redis 中的用户画像快照。

---

### 全流程总览图

```
前端 POST /api/chat
  │
  ▼
ChatController (JWT 认证)
  │
  ▼
ChatApplicationService.submitAsync()
  ├── 请求标准化
  ├── ChatGovernanceService: 配额检查 + 敏感数据检测
  │     ├── 命中 BLOCK → SSE 短路返回
  │     └── 通过 → 脱敏后继续
  ├── Token 配额预留（用户 + Provider）
  └── 发布到 RabbitMQ (chat.requests) + 注册 SSE
        │
        ▼
  ChatRabbitConsumer (异步消费)
        │
        ▼
  AgentOrchestrator → Agent.chat() / chatStream()
        │
        ▼
  ExecutionPipeline (8 个阶段串行执行)
  ┌────────────────────────────────────────────┐
  │  10  ContextEnrichStage     会话历史加载    │
  │  12  UserMemoryRetrieval    用户画像+事实   │
  │  20  ModelResolutionStage   模型解析        │
  │  25  KnowledgeRetrieval     Agent知识库     │
  │  30  MultimodalGuardStage   多模态校验      │
  │  35  AttachmentPreparation  附件URL转换     │
  │  40  ResultAggregationStage 工具+LLM调用    │
  │  50  MemoryPersistStage     记忆事件发布    │
  └────────────────────────────────────────────┘
        │
        ▼
  SSE 推送最终回复给客户端
        │
        ▼
  异步记忆写入 (ChatMemoryPersistConsumer)
  ├── MySQL:  完整会话历史
  ├── Redis:  近期上下文 + 用户画像快照
  └── Milvus: 用户长期事实向量
```

---

### 设计要点

**无状态管线**：`ExecutionPipeline` 本身无状态，所有中间数据通过 `PipelineContext` 传递。多个 Agent 可以共享同一条管线实例，线程安全。

**关注点分离**：每个 Stage 职责单一，通过 order 值确定执行顺序，新增阶段只需实现 `PipelineStage` 接口并设定 order，不需要修改管线本身。

**事件驱动解耦**：聊天请求和记忆写入通过 RabbitMQ 解耦，主链路不等记忆写完就返回，降低响应延迟。

**治理贯穿全链路**：入队前有配额检查和敏感数据脱敏，模型调用前有 Provider 预算检查，响应后有二次敏感数据检测和使用量记录，失败时触发告警。

**全链路可观测**：每个阶段和关键操作都创建 OpenTelemetry Span，通过 traceId 串联全链路，可在 Jaeger 中查看完整的请求处理轨迹。
